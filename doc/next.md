## Sweep 1：S0（优先）——把 Phase3“符号迁移差距”压下来，解决 P1

**目的**：在不重新训练 KAN 的情况下，先把“符号公式 vs KAN”差距缩小（transfer gap），并解决 medium 库出现 inf/NaN 的数值稳定问题（你们已明确这是卡点之一）。

**做法（可执行）**：以当前最强 KAN 作为固定 teacher：

* teacher KAN：`2026-03-01_151000_kan_nobase_nogrid_gpu`（target=`delta_net_load_h6`）。
* 对同一 teacher 跑一组 Phase3 提取 sweep（只变 Phase3 参数）。

建议的 Phase3 网格（9 个组合，够用且不爆炸）：

* `lib = strict`，`r2_threshold ∈ {0.98, 0.99, 0.995}`
* `lib = medium`，`r2_threshold ∈ {0.98, 0.99, 0.995}`
* `lib = strict` +（**提高多项式阶/加 rational 的“稳健增强版”**）`r2_threshold ∈ {0.99, 0.995, 0.999}`

  * 如果你们现有实现没有“strict+增强版”这个 lib，那就先用 strict（纯配置落地），把“增强版”作为**小改动**加入（只改符号库列表，不动训练/评估链路）。

**关键落地点（解决 NaN/inf 的最低成本手段）**：借鉴引用论文对稳定训练的处理：他们强调把特征统一缩放到 [0,1]（Min-Max）以提高稳定性与收敛速度。
你们当前是 Z-score（且 target 不归一化）。
因此对“会爆”的函数（exp/gaussian），**最划算的工程策略是：只在 Phase3 eval 时做安全裁剪/安全函数**，而不是重做数据管线：

* `safe_exp(x) = exp(clip(x, -c, c))`（c 取 8~12 常见就够）
* `safe_div(a,b) = a/(b+eps)`
* 对所有输入先 clip 到训练集分位区间（比如 [p1,p99]）

这一步即使需要加代码，也非常局部（只影响公式 eval，不影响 KAN/MLP 训练），风险低，收益大：medium 库就不该再因为数值爆炸“不可用”。

**验收标准（写进 sweep 的选型规则）**：

* 公式 test RMSE 目标：至少把 2388.46 拉到 **<2000**（经验上这样 skill 会明显上来）。
* transfer gap：`RMSE_sym - RMSE_KAN` 明显下降（你们有 transfer gap 图产物）。
* 公式稳定性：`formula_eval_test.json` 不出现 NaN/inf（你们现在就是用它做复算校验）。

> 为什么先做 S0：因为你现在已经有一个“能赢的 KAN”，但 Phase3 把它“翻译坏了”。只要把翻译质量提上去，论文立刻更像样。

## Sweep 2：S1×S2（合并跑）——用“多步 horizon + 替代关系消融”逼物理因子进公式，解决 P2/P4

你们现在遇到的核心结构性矛盾是：**自回归（lag）太强，导致物理因子进不来**。
而引用论文之所以能在风/光任务里让气象变量成为主角，是因为他们做了两件事：

1. 把任务明确为**多步超短期预测**（15–45 min），以支撑调度决策。
2. 用相关性分析挑选“对出力最有用”的气象变量（风速、GHI 等），避免冗余输入稀释有效信号。

你们已经具备多步派生数据集：`delta_{load,net_load,wind,solar}_h{1,6,12,24}`。
也已经把特征组细分为 `meteo_*` 与 `solar_geom/solar_flag`，用于做 “GHI vs 太阳几何” 的替代消融。

**做法（可执行）**：用 `scripts/thesis_sweep_driver.py` 在一个 sweep 里同时跑：

### 2.1 Horizon sweep（S1）

目标列：`delta_net_load_h6 / h12 / h24`（分别对应 30/60/120 min ahead，5min 分辨率）。
直觉：h 越长，`net_load_t` 的“复制能力”越弱，外生因子更容易成为修正项，从而更可能在稀疏化后仍保留。

### 2.2 替代关系消融（S2）

对每个 horizon，跑 4 档特征组组合（只改 include_groups）：

A) **Full**：`meteo_solar + solar_geom + solar_flag + meteo_wind + calendar/periodic + (你的lags组)`
B) **GHI-only**：保留 `meteo_solar`，去掉 `solar_geom/solar_flag`
C) **Geom-only**：保留 `solar_geom/solar_flag`，去掉 `meteo_solar`
D) **Neither**：两者都去掉（对照组，看性能掉多少）

> 你们之所以能这么做，是因为代码里已经把组拆好了；这是最“低成本换最大信息量”的实验。

### 2.3 再加一条“强制物理因子”的关键配置（建议必加）

在同一组 horizon 下，再跑一档：

* **禁止 wind/solar 的自回归 lag**（让模型别再用 `wind_lag_* / solar_lag_*`），只允许 `net_load`/`load` 的 lag + 气象代理（wind_speed/GHI/temp）。
  这条配置是为了解决你们已经明确指出的问题：当前最强模型把 wind_speed/GHI/temp 剪到 0，依赖 wind/solar lag 与太阳高度角等。

如果你们的参数体系支持 “lag_series 选择哪些序列生成 lag”，就把 wind/solar 从 lag_series 里剔除；如果不支持，就在 include_groups 里去掉对应 lag 子组（或临时在特征构造里屏蔽 wind_lag/solar_lag 列）。目标是一样的：**把可用的“捷径”关掉**。

**验收标准**：

* 预测不要求超过当前最强 KAN（那条可以作为“Accuracy upper bound”）。
* 但要求：

  1. 仍然要有正的 skill（相比 persistence 有改进）。
  2. `feature_importance` 里 `wind_speed_10m_m_s / ghi_w_m2 / temp_2m_c` 至少有 1–2 个进入（不再全为 0）。
  3. Phase3 strict 公式里能显式出现这些变量（哪怕是“修正项”）。

这一步同时解决：

* P2：物理因子进不来
* P4：论文叙事对不齐引用论文（因为你真的做了 multi-step + 特征筛选/消融，跟论文方法论一致）

## Sweep 3：S3（论文叙事最强）——结构化分解：先预测 wind/solar/load，再组合 net_load，解决 P2/P4 并“更像在借鉴论文”

这是你们文档里已经点名的“达标路线”：

> 若论文硬要求“风速、光照强度、温度、历史负荷”显式出现，仍需要通过 S2 或 S3 来达标。
> 而且你们已经有组合脚本 `scripts/combine_net_load_runs.py`，能把 load/wind/solar 三条子模型预测组合成统一评估的 net_load run。

**为什么 S3 是“借鉴引用论文的最佳姿势”**：
引用论文的任务对象就是 **wind power / PV power forecasting**，并通过 Pearson 相关性选出风速与 GHI/DNI/TSI 等作为输入。
你如果直接在 net_load 上引用它，会显得“任务不对口”；但如果你把你的系统拆成 wind/solar 子任务，就和论文高度一致，引用就自然很多。你们自己也明确“不建议替换主线数据集”，而是把论文作为架构/训练策略参考来源。

**做法（可执行）**：选定一个 horizon（建议先从 h=6 起，因为你们已经有最佳基线可对照），训练三条子模型并符号化：

1. **Wind 子模型**：`delta_wind_h6`

   * 输入特征：优先只留 “风速类 + 少量周期项/滞后”，并刻意让 `wind_speed_10m_m_s` 成为主解释变量。
   * 参考论文：风电任务中风速变量与出力相关性 >0.7，最终选择风速作为主要输入。
   * 你们数据里已有 `wind_speed_10m_m_s` 以及工程代理 `wind_speed_10m_m_s_cubed` 等，可用来“用简单库表达强非线性”。

2. **Solar 子模型**：`delta_solar_h6`

   * 输入特征：`ghi_w_m2`（核心） + `solar_altitude/is_night`（物理约束/几何 gating）。
   * 参考论文：PV 任务中 GHI/DNI/TSI 与出力相关性较强，选其为输入。

3. **Load 子模型**：`delta_load_h6`

   * 输入特征：`temp_2m_c`（或 HDD/CDD）+ `load` 的若干 lag + 周期编码。
   * 这条线会很容易写出“温度敏感性”的可解释关系（你们已有 physics_mapping 检查框架）。

然后用组合脚本把三者合成：

* 组合形式（推荐写法）：
  [
  \widehat{net_load}*{t+h} = \widehat{load}*{t+h} - \widehat{wind}*{t+h} - \widehat{solar}*{t+h}
  ]
  或者在 delta 口径：
  [
  \widehat{\Delta net_load}*{h} = \widehat{\Delta load}*{h} - \widehat{\Delta wind}*{h} - \widehat{\Delta solar}*{h}
  ]
  再用你们现有重建逻辑恢复到绝对值评估口径。

**验收标准**（论文导向）：

* 每条子公式必须显式包含目标物理变量：

  * wind 公式里必须出现 `wind_speed_10m_m_s`（最好还能体现非线性：立方/饱和）。
  * solar 公式里必须出现 `ghi_w_m2`（并受 `is_night` 或太阳高度调制）。
  * load 公式里必须出现 `temp_2m_c`（或 HDD/CDD）与历史负荷项。
* 组合后的 net_load 公式天然包含 “风速/光照/温度/历史负荷”（因为来自三个子公式），从而满足你们的硬性论文要求。

> 这一条基本是“最稳的达标路线”：哪怕 direct net_load 的符号提取仍不够好，wind/solar 子任务的物理关系通常更强、更容易提取出高质量公式。

## 如何更好地“借鉴/引用”那篇论文：建议用“可迁移做法映射表”，而不是照搬模型结构

你们文档里已经说得很对：不建议把主线数据集替换成论文数据集，因为任务对象不同（论文是 WP/PV 出力，你们是耦合负荷/净负荷）。
因此更好的借鉴方式是：把论文拆成“可迁移做法”，然后在你的系统里实现“可解释版本”，并在论文里清晰写出“借鉴点—改造点—原因”。

下面给一个你可以直接塞进论文方法章节的映射表（建议做成表格）：

1. **多步预测的任务动机与设置**

* 论文：强调 multi-step 把预测从 15min 扩展到 30/45min，以支持调度与备用分配。
* 你：使用 5min 数据构造 `h={6,12,24}` 的多步目标（30/60/120min），并用你们已实现的 `derive_dataset --horizon-steps` 产生 `delta_*_h{n}`。

2. **相关性驱动的特征筛选**

* 论文：用 Pearson 相关性挑选高相关变量；风电选风速、光伏选 GHI/DNI/TSI，并给出热力图与公式。
* 你：在 S3 中对 wind/solar 子任务复刻这个流程（哪怕你只有一个高度风速，也能画热力图、写“为何选择 wind_speed 与 GHI”），并作为“物理因子入公式”的证据链。

3. **数据缩放提升稳定性**

* 论文：强调 Min-Max 把特征缩放到 [0,1]，提高稳定性与收敛速度。
* 你：主训练仍用 Z-score（已实现）；但在 Phase3 公式评估时对敏感函数采用安全裁剪/安全函数（S0 的关键落地），用论文来解释“为何这样能避免数值不稳定”。

4. **多尺度思想**

* 论文：MCKAN 用多尺度卷积/池化捕捉不同时间尺度的动态变化。
* 你：在可解释约束下，用 multi-scale lag（你们已用 `lag_steps=[12,24,48]`）+ horizon sweep +（可选）滚动统计特征，形成“可解释的多尺度”。

5. **模块消融与实验组织**

* 论文：对 MCKAN/EAA/CQALA 做消融，评估各模块贡献。
* 你：对应消融成 S0–S3：

  * S0：符号库/阈值对“可解释翻译质量”的贡献
  * S1：horizon 对外生因子可解释性的贡献
  * S2：GHI vs 太阳几何替代关系
  * S3：结构化分解对“物理因子显式出现”的贡献
    这会让你的论文结构更像高质量工程论文：每个模块都有实验支撑。
