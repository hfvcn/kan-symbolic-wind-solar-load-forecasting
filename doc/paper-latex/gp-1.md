## Part 1 [The Review Report]

### Summary

该文聚焦于一个有研究味道、且目前文献中确实少被正面处理的问题：当 KAN 的符号提取被用于含自回归滞后特征的净负荷时间序列预测时，公式会系统性坍缩为“只剩滞后项”，作者试图用阻断实验论证其主要原因是自回归捷径竞争，并提出结构化分解作为补救路径。

### Strengths

1. **问题定义是有价值的。** 这篇稿子的真正贡献不在于“又做了一个净负荷预测器”，而在于把“高精度 teacher 无法转化为保留物理量的符号公式”单独拎出来研究。这个问题对 KAN / symbolic regression 在真实时序场景中的落地很关键，社区里也确实缺少这种机制层面的分析。

2. **证据链有递进性，不是单点 anecdote。** 文章不是停留在“我们发现公式坍缩了”这一层，而是从 direct 失败、到 solar/wind 的定向诊断、再到 blocking 干预、最后到 S3 结构化补救，叙事链条完整，且作者对边界条件的描述比较克制，这一点明显好于很多只堆表格的应用论文。

3. **提出了适合本问题的可辨识性指标，并给出建设性方案。** VER/FAR/TGR 这套指标至少把“变量有没有真正进入公式”从主观判断变成了可量化对象；S3 也不是空泛讨论，而是给出了一个可执行的结构化解决方向。这对后续同类工作是有参考价值的。

### Weaknesses (Critical)

1. **[致命] 最核心的机制证据在 Case 4 上并没有做到严格同层对照。** 文中最想站住的结论，是“自回归捷径竞争而非变量无关性导致物理量消失”。但 Case 4 的对照并不干净：未阻断侧用的是“9 组 direct symbolic 配置”的 VER=0/9，而阻断侧用的是“5 个 seed 的活跃边统计”得到 4/5；也就是说，比较对象同时混杂了**symbolic 配置 vs. 随机种子**、**公式层 VER/FAR vs. 结构层 active edges** 两个维度。这样的比较能说明“阻断后情况变好了”，但还不足以支撑文中这么强的机制性表述。更进一步，Case 4 的 blocked 设置仍保留了 load lag，而 direct collapse 恰恰是 load-lag 主导；当前协议无法区分“真正的主导捷径究竟是 load lag、本身的 lag 总量，还是特征预算竞争”。

2. **[致命] S3 目前没有证明“同一个对象同时兼具精度与公式级可解释性”。** 文章自己已经诚实承认：当前 S3 的结果是“三条局部公式 + 一个结构化组合预测器”，还不是显式合成并在 test 上逐点复算过的净负荷总公式。问题在于，正文又多次把它表述成“实现了精度与可解释性的统一”。这在严格审稿里是站不住的，因为目前的高精度来自组合预测器，而可解释性来自局部公式；两者不是同一个 artifact。更麻烦的是，子任务 TGR 分别达到 2.569、1.401、2.302，说明符号公式对 teacher 的逼近损失并不小。也就是说，现有结果更准确的表述应是：**S3 恢复了局部公式可辨识性，并保持了 teacher-level 组合预测精度**；但尚未证明**公式级组合预测**也保有同等精度。

3. **[重要但可修] 实证严谨性距离通用顶会标准还有明显差距。** 如果按 ICLR/NeurIPS/ICML 的主会标准看，baseline 套件明显偏薄：只有参数匹配 MLP 和 direct PySR，而缺少更强的时序/表格基线，也缺少非 KAN 的可解释基线。与此同时，KAN 相对 MLP 的提升并不大，但显著性检验给出极小 p 值；从文稿描述看，这更像是基于大量时间点的逐点误差配对检验，在强自相关测试集上，这种 p 值很容易显著，却未必代表方法层面有稳定、可复现的优势。再叠加单数据集、部分实验仅 5 个种子、以及“插值先于划分”的轻微泄漏，当前证据更像是一篇**很有潜力的机制研究初稿**，而不是已经打磨到顶会接收线的实证论文。

4. **[非致命但会伤害可信度] 文稿中有若干协议层面的不一致，说明实验规范尚未完全冻结。** 比如前文写“26 维输入特征”，后文又写“28 个输入特征”；讨论部分用 load_lag_1 来解释自相关主导，但实际特征定义里是 lag_12/24/48；结果表里出现“扩展”函数库，但方法部分没有把它定义清楚。这些都不是方法硬伤，但它们会让审稿人怀疑：实验协议是否在写作过程中发生过漂移，而一些表述没有被同步更新。

### Rating

**按多个会场口径估计：ICLR/NeurIPS/ICML 主会约 5/10；AAAI/IJCAI 主会约 6/10；更偏应用/能源智能的强会场可到 6.5-7/10。** 评分依据很直接：问题选得准、研究链条有价值，但当前最关键的两块——**Case 4 的机制证据闭环**与 **S3 的公式级精度闭环**——都还没有真正收口，因此距离通用顶会接收线还差一轮决定性的实验。

## Part 2 [Strategic Advice]

### 问题根源

这篇稿子最深层的问题，不是“实验不够多”，而是**论文想证明的对象和实际量到的对象还没有完全对齐**。

第一，关于“捷径竞争”这条主结论，作者想做的是机制判别，但目前在最关键的 Case 4 上，比较层级并不统一：一边是 symbolic config 级别的 collapse，另一边是 seed 级别的 active-edge 恢复。这个问题的根源不是统计技巧，而是实验设计没有在一开始就把“单位分析对象”冻结下来。你需要决定：整篇论文到底以 **seed** 为基本单位，还是以 **symbolic configuration** 为基本单位；到底以 **公式层变量出现** 为主指标，还是以 **剪枝后活跃边** 为主指标。现在是两者混用，所以证据链天然会松。

第二，S3 的问题根源也不是“还差一个表格”，而是**accuracy artifact 和 interpretability artifact 不是同一个对象**。你现在拿到的是：局部公式有可解释性，组合预测器有精度。但顶会审稿人关心的是：最终可交付物究竟是什么？如果最终可交付物是“显式净负荷公式”，那你必须给它单独报 test 复算精度；如果最终可交付物其实是“结构化 teacher 组合 + 附带局部公式解释”，那论文定位就不能再写成“公式级统一”。

第三，baseline 薄弱的根源是**论文定位摇摆**。现在这篇稿子一半像 forecasting paper，一半像 interpretability/mechanism paper。结果就是两边都没打满：按 forecasting 标准，强 baseline 不够；按 mechanism 标准，干预设计还差最后一步收紧。你必须在投稿前明确：究竟主打“机制研究”，还是主打“应用方法”。两者的实验重心完全不同。

第四，文稿中的 26/28 维、load_lag_1、扩展函数库未定义等问题，本质上反映的是**研究日志已经收敛，但论文协议文本还没有完全收敛**。这类问题单独看不致命，但会显著削弱审稿人对实验冻结程度的信任。

### 可救性判断

好消息是，这篇稿子的多数问题是**可在修订期内补强**的，而且补强后说服力会明显上一个台阶。

最容易救、也最该优先救的是 Case 4。你只需要把 blocked / unblocked 做成**完全匹配的 paired protocol**：同一组 seeds、同一套 symbolic 库、同一套 pruning 规则、同一层指标，同时报 VER、FAR、edge count、TGR、RMSE。只要这一版结果还支持原结论，你的主张就会比现在硬得多。这是高优先级、强可修项。

S3 也有机会救，但分两种情况。如果你能把
(F_{\text{net}} = F_{\text{load}} - F_{\text{wind}} - F_{\text{solar}})
显式合成出来，并在 test 上逐点复算后仍保持接近当前 composite 的 RMSE/Skill，那么这篇稿子的档次会明显上升，因为你就真正完成了“公式级闭环”。但如果一旦显式公式复算，性能明显掉下去，那就说明这不是写作问题，而是**当前方法本身还不能同时满足精度与公式级可解释性**。这属于方法层面的结构性边界，不能靠修辞补过去。此时最正确的做法不是硬扛，而是主动降级 claim，把贡献收缩为“恢复局部公式可辨识性，并给出结构化预测器”。

baseline 问题也可修，但投入产出比要控制。你不需要把所有时序模型都打一遍；你只需要补到能让审稿人闭嘴的程度：一个强表格/树模型，一个合理的时序深度基线，一个非 KAN 的解释性基线，就够了。这个问题是工程量问题，不是结构性死局。

真正比较难靠补实验完全解决的，是“为什么偏偏是 shortcut competition，而不是符号库表达能力不足、剪枝偏置、或任务分解改变了问题难度”。这类替代理由很难被一次 rebuttal 全部排空，所以你在主文和 rebuttal 里都应该主动收缩表述：不要写“证明了唯一机制”，而要写“在当前 KAN 提取链路和数据处理设置下，blocking 提供了支持 shortcut competition 的干预性证据”。这会安全得多。

### 行动指南

最优先要做的，不是继续加新故事，而是把现在这条主线彻底收口。

第一步，**重跑一个真正冻结的 canonical intervention table**。针对 direct net load，至少给出三组严格匹配的条件：保留全部 lags、只保留 load lag、移除全部 lags。每组用同样的 5 或 10 个 seeds、同一 symbolic 库、同一阈值、同一评估口径，统一报告 teacher RMSE、symbolic RMSE、VER、FAR、edge count、TGR。这样你才能回答三个关键问题：到底是谁在形成捷径；恢复的是“活跃边”还是“进入公式”；精度损失究竟有多大。没有这张表，顶会审稿人一定会抓住 Case 4 不放。

第二步，**把 S3 的“组合预测器”与“组合公式”彻底拆开写清楚**。你需要同时给出 teacher-composite 和 symbolic-composite 两个对象：前者说明结构化分解是否提升预测，后者说明公式级闭环是否成立。若 symbolic-composite 明显劣化，也不要藏，直接把它写成结果：这反而会让论文更可信。当前最危险的写法，是把 teacher-composite 的精度和 local formula 的 FAR 并列后，暗示已经完成统一。

第三步，**补最少但足够的 baseline**。如果走通用 ML/AI venue，建议至少补一个 LightGBM/XGBoost 这类强表格基线，加一个轻量时序基线（TCN/LSTM/Transformer-lite 任一即可），再加一个解释性基线（例如 GAM/EBM 或者分解子任务上的 PySR/线性 ARX）。这三类 baseline 分别对应三个质疑：是不是 KAN 才能做好预测、是不是时序深度模型本来就更强、是不是“可解释性”只能靠 KAN 才恢复。

第四步，**全面清理协议文本**。把 26/28 维特征统一；把 load_lag_1 的说法改成与你真实特征一致的 lag_12/24/48，或者明确说明讨论里是概念性举例；把“扩展”函数库的定义补到方法里；把所有 blocked/unblocked 的评估口径写死。这类问题不会帮你加分，但不修一定掉分。

第五步，**按会场调整叙事中心**。如果投 ICLR/NeurIPS/ICML，就把题目和摘要进一步往“机制研究”收：主打 formula collapse、shortcut competition、interventional evidence、structured remedy，弱化 forecasting SOTA 口吻。如果投 AAAI/IJCAI 或更应用导向会场，就在保留机制贡献的同时补强实际预测 baseline 和工程意义。现在这篇稿子最大的风险不是内容不够，而是定位太宽，导致每个方向都差半步。

最后，若进入 rebuttal 场景，最该主动降攻击面的表述是两句：第一句，不再把 S3 写成“已经得到高精度净负荷公式”，而写成“得到局部公式并构成结构化预测器”；第二句，不再把 shortcut competition 写成“被完全证明的唯一原因”，而写成“在当前 KAN-to-symbol pipeline 下得到干预支持的主导解释”。这两处一旦收紧，你的稿子会从“claim 过强”变成“结论稳健”。
